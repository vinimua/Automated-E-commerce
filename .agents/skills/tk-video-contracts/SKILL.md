# TK Video Contracts

This project is contract-first. Before implementing or changing behavior that crosses service boundaries, read the relevant contract file in `docs/`.

## Contract Sources

- Database and main state source: `docs/01-database-schema.sql`
- Java API and callbacks: `docs/02-openapi-spec.yaml`
- Python AI structured outputs: `docs/03-ai-output-json-schema.md`
- Python-to-Render Worker render protocol: `docs/04-render-manifest-schema.md`

## Required Workflow

1. Identify which service boundary is affected.
2. Read the relevant contract file before editing code.
3. Do not add fields, statuses, enum values, or callback shapes in code unless the corresponding contract is already updated or updated in the same change.
4. Keep database, OpenAPI, AI Schema, and RenderManifest names aligned.
5. If changing a contract, check all downstream services that consume it.

## Non-Negotiable Invariants

- Java Spring Boot + PostgreSQL is the only main business state source.
- Frontend calls only Java API. It must not call Python AI service or Render Worker directly.
- Python AI service must not write core Java business tables directly.
- Render Worker must not understand users, quotas, products, or task business state. It only consumes `RenderManifest` and calls Java callback APIs.
- `videoType` is the business video type, for example `pain_point_solution`.
- `template` is the render template, for example `pain_point_solution_v1`.
- AI callbacks use `schemaVersion + stage + status`.
- RenderManifest uses `manifestVersion = "1.0.0"`.
- Failure payloads use `errorCode`, `errorMessage`, `failedStage`, and `retryable`.

## Common Checks

- Task status must follow the status flow documented in `01-database-schema.sql`.
- Callback handlers must be idempotent.
- Quota consume/refund/grant records must use idempotency keys.
- Failed callbacks must not require success-only fields such as `videoId` or material `url`.
- JSON generated by AI must pass schema validation before callback.
